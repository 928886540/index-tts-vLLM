## 当前定位

这个项目不是云服务，不做 SaaS、多用户账号、计费、公网认证后台。

目标是把项目交付给用户，让用户在自己的电脑上安装和启动。本机服务能用，用户自己的机器也能用。

所以架构重点不是“多人并发”，而是：

- 同一台机器里多条 Tavo 消息可能同时点击播放。
- LLM 解析可以先并行完成。
- TTS 生成应该受控排队，避免多个推理互相抢 GPU。
- 只有当前播放卡片适合走流式播放。
- 非当前卡片如果排队生成完成，应该只保存落盘，不要突然抢播。
- 如果已有任务在运行，可以提示用户等待音频自动落盘。

## 建议补齐的实现点

1. 增加本机 TTS 任务队列。
   - 简单 FIFO 即可。
   - 不需要 Redis / RabbitMQ。
   - 先保证同一时间只有一个重型 TTS 推理任务，或按显存允许限制并发数。

2. 拆分任务阶段。
   - `llm_parsed`：LLM 已拆段。
   - `queued`：等待 TTS。
   - `streaming`：当前卡片正在流式播放。
   - `rendering_to_cache`：后台生成并落盘，不抢播。
   - `cached`：已落盘，可直接播放。
   - `failed`：失败。

3. 前端播放策略。
   - 当前用户点的卡片：允许流式播放。
   - 其它卡片：只排队生成，完成后写 cache / IndexedDB，状态显示“已生成”。
   - 如果用户切换播放另一张卡片，明确停止或降级上一张卡片的流式，不要两个卡片同时争播放。

4. 用户提示。
   - 已有 TTS 任务运行时，不要让用户以为卡死。
   - 提示“已有音频任务运行中，本条已排队，生成完成后会保存音频”。

这版图的重点就是把最初设计图里偏云服务的部分，改成现在实际需要的本机交付架构。
